# How my-ssg Works

This site was generated by **my-ssg** — a static site generator built with a TypeScript CLI and a Go engine. Here's a quick look at how it all fits together.

## Architecture

The project is split into two layers that work together:

| Layer | Language | Responsibility |
|---|---|---|
| CLI | TypeScript | Parses commands, finds the correct binary |
| Engine | Go | Reads Markdown, renders templates, writes HTML |

When you run `my-ssg build <directory>`, the TypeScript layer hands off to the Go engine, which does the heavy lifting.

## The Build Pipeline

Every Markdown file goes through four steps:

1. **Parse** — the YAML front matter is extracted and decoded
2. **Convert** — the Markdown body is converted to HTML via [goldmark](https://github.com/yuin/goldmark)
3. **Render** — the HTML is injected into the Go `html/template` layout
4. **Write** — the final file is saved to the `dist/` directory

## Supported Markdown

Because the engine uses goldmark with GFM extensions, you get full GitHub-Flavored Markdown support out of the box:

- **Bold**, *italic*, ~~strikethrough~~
- `inline code` and fenced code blocks
- Tables (like the one above)
- Task lists, footnotes, and autolinks

Here's an example code block:

```typescript
import { Command } from 'commander';

const program = new Command();

program
  .command('build')
  .argument('<directory>')
  .action((dir) => buildSite(dir));

program.parse();
```

## Front Matter

Each `.md` file starts with a YAML front matter block that controls how the page is built:

```yaml
---
title: "My Post"
date: 2026-01-15
description: "A short summary"
draft: false
slug: my-post
---
```

Set `draft: true` to exclude a page from the build — useful for work in progress.

## CI/CD

Every push to `main` triggers a GitHub Actions pipeline with five checks:

- ✅ Go Engine Build
- ✅ Go Tests
- ✅ TypeScript CLI Build
- ✅ TypeScript Tests
- ✅ Deploy to GitHub Pages

The site you're reading right now was deployed automatically by that pipeline.
